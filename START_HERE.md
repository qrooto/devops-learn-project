# С чего начать и как работать с курсом

## Порядок чтения файлов

```
START_HERE.md        ← ты здесь
    │
    ▼
CURRICULUM.md        ← читай раздел текущего уровня перед запуском
    │
    ▼
level-N/README.md    ← пошаговые команды для выполнения
    │
    ▼
level-N/код          ← смотри только когда выполняешь конкретный шаг
```

**Перед началом прочитай `INFRASTRUCTURE_PLANNING.md`** — там разбор сколько серверов нужно, какие требования у каждого уровня и что делать с GitLab.

**CURRICULUM.md и README.md — разные вещи.**
- `CURRICULUM.md` — учебник: объясняет зачем, как устроено, анатомия файлов. Читай до запуска.
- `level-N/README.md` — инструкция: конкретные команды в нужном порядке. Выполняй шаг за шагом.

---

## Протокол прохождения уровня

Это основной режим работы. Смысл: README уровня — это ключ к задачнику, а не пошаговый рецепт. Опыт даёт только самостоятельное построение, README нужен для сверки.

1. `[голова]` **Прочитай постановку.** Из README уровня — только постановку задачи и «боль». README закрыть.
2. `[руки]` **Построй сам.** По официальной документации (таблица ниже в «Что читать параллельно»), не подглядывая в готовые файлы уровня.
3. `[руки]` **Сломай и почини.** Раздел «Что сломать намеренно» + всё что сломалось само. Диагностика — по лестнице дебага (ниже).
4. `[голова]` **Сверься.** Открой README и готовые файлы, сделай diff со своим решением. Каждое расхождение — вопрос: «это важно или вкусовщина?». Важные расхождения — разобрать до конца.
5. `[голова]` **Запиши.** 3–5 строк в ЖУРНАЛ в своём локальном PROGRESS.md (см. «Как проходить» в README) и следующее действие в СЕЙЧАС.

### Типы задач: [голова] и [руки]

Задачи размечены под два типа сессий (пометка есть и в начале README каждого уровня):

- **[голова]** — чтение постановки, планирование, diff и разбор расхождений, теория, «На собеседовании спросят», Security Block. Не требует терминала — подходит для коротких сессий в дороге, offline.
- **[руки]** — hands-on: команды, дебаг, «сломай и почини». Требует VM и время на погружение — домашние сессии.

Короткая сессия в метро ≠ потерянная: возьми [голова]-задачу текущего уровня (прочитать постановку следующего шага, разобрать вчерашний diff, прогнать вслух вопросы собеседования).

Детальный разбор шагов из README (раздел «Как работать с каждым уровнем» ниже) используй на этапе сверки — или как запасной режим, если уровень совсем не идёт.

### Лестница дебага

Когда что-то сломалось — эскалируй строго по ступеням. На следующую ступень — только когда текущая исчерпана. На ступенях 1–2 таймбокс ~30–45 минут: раньше не сдавайся, дольше не закапывайся.

1. **Ошибка и логи целиком.** `docker logs <container>`, `kubectl describe <pod>` → секция Events, `kubectl logs --previous`. Прочитай сообщение об ошибке полностью, не первые слова.
2. **Проверка гипотез руками изнутри.** `docker exec` / `kubectl exec` → `env`, `curl` до соседнего сервиса, `nslookup` / `getent hosts`. Проверяй что реально видит процесс, а не что ты думаешь он видит.
3. **Официальная документация** по подозреваемой опции/директиве. Точечно, не с начала.
4. **Поиск по точному тексту ошибки** (в кавычках, без своих путей/имён).
5. **AI в режиме наставника** — наводящие вопросы, а не «почини за меня» (режимы описаны в CLAUDE.md).
6. **README уровня как ключ к задачнику.** Факт эскалации сюда зафиксируй в ЖУРНАЛЕ — это сигнал, что тема требует повторения.

## Как работать с каждым уровнем

**1. Открой CURRICULUM.md → найди раздел уровня**

Прочитай:
- "Зачем это нужно" — убедись что понимаешь какую боль решаем
- "Анатомия файлов" — разбери каждую директиву до запуска, не после
- "Как это работает" — прочитай аналогию, она поможет когда будет непонятно

**2. Открой level-N/README.md**

Читай и выполняй команды по одному шагу. Не беги вперёд — каждый шаг опирается на результат предыдущего.

**3. Смотри в код когда README просит**

Не читай весь код сразу — только те файлы которые упомянуты в шаге. Иначе информации слишком много и она не укладывается.

**4. Выполни раздел "Что сломать намеренно"**

Это не опциональная часть. Диагностика сломанного — главный практический навык DevOps. Пропускать нельзя.

**5. Пройди Security Block и Best Practices Checklist**

В конце каждого уровня три дополнительных блока:
- **Security Block** — принципы безопасности применённые на уровне, почему именно так, антипаттерны (1-2 примера как нельзя делать и почему)
- **Best Practices Checklist** — список для самопроверки. Пройдись и убедись что всё отмечено.
- **Troubleshooting** — 5 типовых проблем для технологий этого уровня. Команды диагностики с объяснением что они показывают.

**6. Запиши инцидент**

После каждой поломки (намеренной или нет) — одна запись в личный файл `incidents.md`:
```
## <дата>: <что сломалось>
Симптом: что видел в терминале
Что сделал: какие команды запустил для диагностики
Причина: что оказалось на самом деле
Решение: что помогло
```

**6. Ответь на вопросы из "На собеседовании спросят" вслух**

Не про себя — вслух. Если не можешь объяснить за 30 секунд — есть пробел который нужно закрыть прежде чем идти дальше.

**7. Сделай коммит**

```bash
cd ..
git add level-N/
git commit -m "level-N: краткое описание что сделал"
git push origin main
```

## Что читать параллельно

Курс намеренно не объясняет всё — только то что нужно прямо сейчас. Когда встречаешь непонятную директиву которой нет в анатомии:

| Инструмент | Документация |
|-----------|-------------|
| Docker | docs.docker.com/reference/dockerfile |
| Docker Compose | docs.docker.com/compose/compose-file |
| nginx | nginx.org/en/docs |
| Kubernetes | kubernetes.io/docs/reference |
| Helm | helm.sh/docs |
| Terraform | developer.hashicorp.com/terraform/language |
| Ansible модули | docs.ansible.com/ansible/latest/collections/ansible/builtin |
| GitHub Actions | docs.github.com/en/actions |
| Prometheus | prometheus.io/docs/prometheus/latest/configuration |

Открывай документацию точечно — только для конкретной директивы которая непонятна. Не читай документацию с начала, это потеря времени.

## Признаки что что-то пошло не так

**Ты читаешь не запуская команды** → стоп. Открой терминал.

**Ты запускаешь команды не понимая зачем** → стоп. Вернись к CURRICULUM.md, прочитай "Зачем" и "Анатомию".

**Ты пропустил "Что сломать намеренно"** → вернись и сделай. Без этого следующий уровень будет на песке.

**Ты застрял на K8s больше 3 дней** → нормально. Это самый сложный уровень. Сделай перерыв, перечитай "Ключевые концепции" в CURRICULUM.md с нуля.

**Что-то сломалось не по плану** → хорошо. Это реальный DevOps. Диагностируй по алгоритму: логи → статус процессов → ресурсы → сеть. Запиши в incidents.md.
