# DevOps Learning Project: Доска объявлений

## Контекст

Я изучаю DevOps на практике. Мой текущий уровень — базовое понимание Docker.

Мы будем строить **доску объявлений** (фронт, бэк, база данных, файловое хранилище для картинок) и постепенно развивать инфраструктуру от минимальной до production-ready с Kubernetes и CI/CD.

---

## Стек приложения и почему именно Go для бэкенда

Наше приложение состоит из трёх частей:

| Часть | Технология | Язык |
|---|---|---|
| Фронтенд | HTML + Vanilla JS, раздаётся через Nginx | — |
| Бэкенд | REST API | **Go (Golang)** |
| База данных | PostgreSQL | — |

### Почему Go, а не Python или Node.js?

**Go — это язык DevOps-инфраструктуры.** Docker, Kubernetes, Prometheus, Terraform, k6 — всё это написано на Go. Изучая DevOps, ты неизбежно столкнёшься с Go-кодом при чтении документации, конфигов, исходников инструментов.

Конкретные причины выбора для нашего учебного проекта:

1. **Компиляция в один бинарник.** `go build` → получаем один файл без зависимостей. Это идеально демонстрирует разницу между интерпретируемыми и компилируемыми языками и делает Docker-образы минимальными (можно использовать `scratch` — образ буквально пустой).

2. **Маленький Docker-образ.** Go-бинарник в `scratch`-контейнере весит ~15-20 МБ. Python Flask-приложение в alpine — уже 80-150 МБ. Это будет хорошо видно на практике.

3. **Производительность под нагрузкой.** Go обрабатывает конкурентные запросы через горутины (goroutines) — очень дёшево по памяти. При нагрузочных тестах (Уровень 1-2) мы увидим реальную разницу.

4. **Простота чтения.** Go — намеренно простой язык. Базовый синтаксис REST-сервера читается почти как псевдокод.

5. **Multi-stage build — учебный must-have.** Компилируемые языки дают возможность показать многоэтапную сборку Docker: стадия сборки (с компилятором) → стадия запуска (голый бинарник). Это важный паттерн, который используется повсеместно.

### Фронтенд — намеренно простой

Фронт — это HTML + Vanilla JS (fetch API). Никакого React/Vue. Причина: мы учим DevOps, а не фронтенд. Фронт будет лежать в папке `frontend/`, Nginx будет его раздавать как статику.

## Принцип обучения

Каждый уровень строится по одной схеме:

1. **Строим** — поднимаем инфраструктуру этого уровня
2. **Ломаем** — запускаем нагрузочные тесты или искусственно создаём проблему
3. **Наблюдаем** — смотрим что происходит, читаем логи, метрики
4. **Чиним/улучшаем** — добавляем новую технологию которая решает боль
5. **Переходим на следующий уровень**

Каждый шаг должен сопровождаться:
- Объяснением **зачем** мы это делаем
- Конкретными командами которые я запускаю
- Описанием **что я должен увидеть** после выполнения
- Заданием для самопроверки

## Уровни инфраструктуры

### Уровень 1 — Монолит (старт)
**Стек:** Docker Compose, Nginx, один Go-контейнер, PostgreSQL

**Что строим:**
- Фронтенд (статика, раздаётся через Nginx)
- Бэкенд на Go (REST API, один инстанс)
- PostgreSQL
- Nginx как веб-сервер

**Ключевой момент с Go:**
- Пишем минимальный Go REST API (~80 строк)
- Смотрим `go build`, понимаем что такое компиляция
- Собираем multi-stage Dockerfile: builder (golang:alpine) → runner (scratch)
- Сравниваем размер образа с Python-аналогом

**Как ломаем:**
- Нагрузочный тест через `wrk` или `k6` на API
- Смотрим как бэкенд задыхается при нагрузке
- Смотрим как растёт время ответа и появляются ошибки

**Боль которую обнаруживаем:** один бэкенд не справляется с нагрузкой

---

### Уровень 2 — Горизонтальное масштабирование
**Стек:** Docker Compose, Nginx как балансировщик, несколько инстансов бэкенда

**Что строим:**
- Поднимаем 3 инстанса бэкенда
- Настраиваем Nginx upstream с балансировкой
- Разбираемся с проблемой сессий (они хранились в памяти одного инстанса)

**Как ломаем:**
- Убиваем один контейнер во время нагрузки (`docker kill`)
- Смотрим падают ли запросы и как быстро восстанавливается

**Боль которую обнаруживаем:** база данных становится узким местом, медленные запросы

---

### Уровень 3 — Кэширование и оптимизация БД
**Стек:** + Redis, + мониторинг запросов к PostgreSQL

**Что строим:**
- Redis для кэширования частых запросов (список объявлений)
- Медленные запросы к PostgreSQL — находим и добавляем индексы
- Смотрим на разницу в скорости до/после

**Как ломаем:**
- Нагрузочный тест на эндпоинт который идёт в базу
- Сравниваем время ответа с кэшем и без

**Боль которую обнаруживаем:** деплой новой версии роняет сервис

---

### Уровень 4 — CI/CD
**Стек:** + GitHub Actions

**Что строим:**
- Pipeline: тесты → сборка Docker образа → пуш в registry → деплой
- Стратегия rolling update в Docker Compose (zero downtime deploy)
- Версионирование образов по git tag/commit

**Как ломаем:**
- Намеренно коммитим сломанный код — проверяем что pipeline падает
- Смотрим как выглядит деплой без zero downtime vs с ним

**Боль которую обнаруживаем:** сложно управлять множеством контейнеров, нет самовосстановления

---

### Уровень 5 — Kubernetes (minikube)
**Стек:** minikube, kubectl, Deployment, Service, Ingress

**Что строим:**
- Переносим docker-compose в Kubernetes манифесты
- Deployments для каждого сервиса
- Services для внутренней коммуникации
- Ingress для внешнего доступа
- ConfigMap и Secrets

**Как ломаем:**
- Убиваем Pod вручную — смотрим как Kubernetes его пересоздаёт
- Нагрузочный тест — смотрим как работает readiness/liveness probe

**Что добавляем:**
- Horizontal Pod Autoscaler (HPA) — автомасштабирование под нагрузкой
- Resource limits и requests

---

### Уровень 6 — Мониторинг и observability
**Стек:** + Prometheus, Grafana, Loki (или ELK)

**Что строим:**
- Prometheus собирает метрики с бэкенда и Kubernetes
- Grafana дашборды: RPS, latency, error rate, CPU/RAM контейнеров
- Централизованные логи через Loki
- Алёрты при деградации сервиса

**Как ломаем:**
- Запускаем нагрузку и смотрим графики в реальном времени
- Симулируем memory leak — наблюдаем как это видно на графиках

---

### Уровень 7 — Продвинутые стратегии деплоя
**Стек:** + Helm, blue-green или canary deployment

**Что строим:**
- Helm chart для всего приложения
- Blue-green деплой: две среды, переключаем трафик мгновенно
- Или canary: выкатываем новую версию на 10% трафика

## Образовательный блок: Языки программирования и DevOps

### Какие языки для чего используются

Это не теория ради теории — понимание этого помогает читать чужие репозитории, выбирать инструменты и понимать почему тот или иной сервис написан именно так.

#### Go (Golang)
**Для чего:** микросервисы, DevOps-инструменты, сетевые сервисы, CLI-утилиты  
**Примеры в дикой природе:** Docker, Kubernetes, Prometheus, Terraform, k6, Consul, Vault, Caddy  
**Почему популярен в DevOps:** компилируется в один бинарник, нет рантайма, отличная конкурентность, простой деплой  
**Как запускать локально:**
```bash
go run main.go          # запустить без компиляции (для разработки)
go build -o app main.go # скомпилировать в бинарник
./app                   # запустить бинарник
```

#### Python
**Для чего:** скриптинг, автоматизация, data science/ML, веб (Django, FastAPI, Flask), DevOps-скрипты  
**Примеры в дикой природе:** Ansible, AWS CLI, многие ML-сервисы, скрипты деплоя  
**Почему популярен в DevOps:** огромная экосистема, быстро пишется, отличен для скриптов и автоматизации  
**Как запускать локально:**
```bash
python3 main.py                  # просто запустить
pip install -r requirements.txt  # установить зависимости
uvicorn app:app --reload         # запустить FastAPI/ASGI сервер
```

#### Node.js / TypeScript
**Для чего:** веб-бэкенд (API), фронтенд (React/Vue/Angular), BFF (Backend For Frontend), real-time приложения  
**Примеры в дикой природе:** многие стартаповские API, Vercel, Netlify функции  
**Почему популярен:** один язык для фронта и бэка, огромный npm-экосистем, асинхронность из коробки  
**Как запускать локально:**
```bash
node index.js            # запустить
npm install              # установить зависимости
npm run dev              # запустить в dev-режиме (обычно с hot-reload)
npx ts-node src/main.ts  # запустить TypeScript без компиляции
```

#### Java / Kotlin
**Для чего:** enterprise-приложения, Android, высоконагруженные сервисы, финтех, банки  
**Примеры в дикой природе:** Netflix backend, Kafka, Elasticsearch, большинство корпоративного ПО  
**Особенность:** требует JVM (Java Virtual Machine) — рантайм, который нужно поставить  
**Как запускать локально:**
```bash
javac Main.java   # скомпилировать .java → .class
java Main         # запустить
# или через Maven/Gradle:
./mvnw spring-boot:run
./gradlew bootRun
```

#### Rust
**Для чего:** системное программирование, производительность-критичные сервисы, WebAssembly, замена C/C++  
**Примеры в дикой природе:** ripgrep, InfluxDB 3.0, части Firefox, Cloudflare Workers  
**Почему важен:** безопасность памяти без GC, производительность как у C, растущая популярность  
**Как запускать локально:**
```bash
cargo run            # скомпилировать и запустить
cargo build --release # собрать оптимизированный бинарник
```

#### PHP
**Для чего:** традиционный веб, CMS (WordPress, Drupal, Laravel)  
**Примеры в дикой природе:** Wikipedia, WordPress сайты, Facebook (исторически)  
**Как запускать локально:**
```bash
php -S localhost:8080   # встроенный dev-сервер
php artisan serve       # Laravel
```

### Интерпретируемые vs Компилируемые: в чём разница для DevOps

```
Интерпретируемые (Python, Node.js, PHP, Ruby):
Исходный код → [Интерпретатор] → Выполнение
                    ↑
             нужен на машине всегда

Компилируемые (Go, Rust, C++):
Исходный код → [Компилятор] → Бинарник → Выполнение
                    ↑                ↑
             нужен только         можно копировать
             при сборке           на любую машину

JVM-языки (Java, Kotlin, Scala):
Исходный код → [Компилятор] → Bytecode → [JVM] → Выполнение
                                              ↑
                                    нужен на машине всегда
```

**Что это значит для Docker-образов:**
- Python/Node: нужно тащить интерпретатор в образ → образы больше (~100-300 МБ)
- Go/Rust: бинарник сам по себе, можно положить в `scratch` → образы маленькие (~10-30 МБ)
- Java: нужен JRE в образе → образы большие (~200-400 МБ), но есть GraalVM native image

### Как строить Docker-образы для разных языков

#### Go — multi-stage build (эталонный пример)
```dockerfile
# Стадия 1: сборка (этот слой в финальный образ не войдёт)
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o server .

# Стадия 2: минимальный образ для запуска
FROM scratch
COPY --from=builder /app/server /server
EXPOSE 8080
CMD ["/server"]
# Итоговый размер: ~15-20 МБ
```

#### Python — обычная сборка
```dockerfile
FROM python:3.12-alpine
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8080
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8080"]
# Итоговый размер: ~80-150 МБ
```

#### Node.js — с multi-stage для production
```dockerfile
# Стадия 1: установка зависимостей
FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

# Стадия 2: сборка (если TypeScript)
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Стадия 3: запуск
FROM node:20-alpine
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY --from=builder /app/dist ./dist
EXPOSE 3000
CMD ["node", "dist/index.js"]
# Итоговый размер: ~150-200 МБ
```

#### Java (Spring Boot)
```dockerfile
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY target/*.jar app.jar
EXPOSE 8080
CMD ["java", "-jar", "app.jar"]
# Итоговый размер: ~200-300 МБ
# Для уменьшения — использовать jlink или GraalVM native image
```

### Ключевые паттерны при сборке образов

**1. Порядок COPY важен для кэша**
```dockerfile
# Правильно: сначала файлы зависимостей, потом исходники
COPY go.mod go.sum ./   # меняется редко → слой кэшируется
RUN go mod download
COPY . .                 # меняется часто → пересобирается только этот слой

# Неправильно: каждый раз будут скачиваться зависимости
COPY . .
RUN go mod download
```

**2. Используй `.dockerignore`**
```
# .dockerignore
node_modules/
.git/
*.log
.env
dist/
```

**3. Не запускай контейнер от root**
```dockerfile
# Добавь в конец Dockerfile
RUN adduser -D appuser
USER appuser
```

**4. Закрепляй версии образов**
```dockerfile
# Плохо — при следующей сборке может сломаться
FROM python:latest

# Хорошо — воспроизводимая сборка
FROM python:3.12.3-alpine3.19
```

### Зависимости: как они устроены в разных языках

| Язык | Файл зависимостей | Lock-файл | Команда установки |
|---|---|---|---|
| Go | `go.mod` | `go.sum` | `go mod download` |
| Python | `requirements.txt` или `pyproject.toml` | `requirements.lock` | `pip install -r requirements.txt` |
| Node.js | `package.json` | `package-lock.json` | `npm install` |
| Rust | `Cargo.toml` | `Cargo.lock` | `cargo build` |
| Java (Maven) | `pom.xml` | — | `mvn install` |

**Lock-файл** — это фиксация точных версий всех зависимостей и их зависимостей. В Docker-образах всегда используй lock-файл, иначе сборка будет непостоянной.
