# Сколько серверов нужно для курса?

## Короткий ответ

**Одного VPS хватит для 90% курса.** Но есть нюансы — и лучше знать о них заранее чем столкнуться в процессе.

---

## Как устроено прохождение курса

Важно понять: уровни не работают одновременно. Ты проходишь уровень 1, изучаешь, ломаешь, переходишь к уровню 2. Каждый уровень — это отдельная папка со своим `docker-compose.yml`. Перед переходом ты делаешь `docker compose down` на текущем.

Это значит что в любой момент времени на сервере работает только **один уровень**, а не всё сразу. Это сильно снижает требования к железу.

---

## Уровни и их требования к ресурсам

### Уровень 0 — Настройка VPS
Минимум. Только базовые пакеты, никаких сервисов.
**1 VPS достаточно.**

---

### Уровень 1 — Монолит
Nginx + FastAPI + PostgreSQL = 3 контейнера.

| Сервис | RAM |
|---|---|
| nginx | ~30 МБ |
| backend | ~80 МБ |
| postgres | ~50 МБ |
| **Итого** | **~200 МБ** |

**1 VPS достаточно. Минимум: 1 ГБ RAM.**

---

### Уровень 2 — Масштабирование
3 инстанса бэкенда на одной машине.

**Честный разговор об этом уровне:**
В реальном production горизонтальное масштабирование = разные физические серверы. То что мы делаем — 3 контейнера на одной машине — это симуляция. Мы изучаем **конфигурацию** (Nginx upstream, балансировка, stateless) а не реальное распределение нагрузки.

Для учебных целей это абсолютно нормально и правильно — концепции те же, разница только в том что CPU и RAM всё равно одна физическая машина.

| Сервис | RAM |
|---|---|
| nginx | ~30 МБ |
| backend × 3 | ~240 МБ |
| postgres | ~50 МБ |
| **Итого** | **~350 МБ** |

**1 VPS достаточно. Минимум: 1 ГБ RAM.**

---

### Уровень 3 — Кэширование
+ Redis.

| Сервис | RAM |
|---|---|
| nginx | ~30 МБ |
| backend × 3 | ~240 МБ |
| postgres | ~50 МБ |
| redis | ~10 МБ |
| **Итого** | **~380 МБ** |

**1 VPS достаточно. Минимум: 1 ГБ RAM.**

---

### Уровень 3.5 — HTTPS (Traefik)
Заменяет Nginx на Traefik, добавляет TLS.

Требует **реальный домен** — Let's Encrypt не выдаёт сертификаты для IP-адресов. Можно купить дешёвый домен (~$1-3/год) или пропустить уровень и вернуться позже.

**1 VPS достаточно.** Требование: домен указывающий на этот IP.

---

### Уровень 4a — CI/CD с GitHub Actions
GitHub Actions runner запускается на инфраструктуре GitHub — их сервера, не твои. Он просто подключается к твоему VPS по SSH для деплоя.

Дополнительные ресурсы на VPS: почти ноль (SSH-соединение во время деплоя).

**1 VPS достаточно.**

---

### Уровень 4b — Self-hosted GitLab

Это особый случай. GitLab CE — тяжёлое приложение.

**Официальные минимальные требования GitLab:**
- 4 ГБ RAM (рекомендуется 8 ГБ)
- 2 CPU
- 10 ГБ диска

Если запустить GitLab на том же VPS где живёт приложение — оба будут страдать. GitLab в одиночку занимает ~2-3 ГБ RAM при старте.

**Варианты:**

**Вариант А (рекомендую): gitlab.com бесплатно**
GitLab.com даёт бесплатные CI/CD минуты и private репозитории. Для изучения GitLab CI синтаксиса и pipeline — полностью достаточно. Self-hosted GitLab нужен когда хочешь контролировать инфраструктуру, для учёбы это избыточно.

**Вариант Б: отдельный маленький VPS для GitLab**
2 CPU, 4 ГБ RAM (~$10-20/месяц на DigitalOcean/Hetzner). Поднять GitLab там, runner на основном VPS или тоже там.

**Вариант В: запустить на основном VPS если у него ≥ 6 ГБ RAM**
Работать будет, но тяжело. GitLab + приложение уровня 2-3 = 3-4 ГБ суммарно.

**Итог: Уровень 4b — единственный где серьёзно рассмотри второй VPS или gitlab.com.**

---

### Уровень 5 — Kubernetes (minikube)

minikube поднимает полноценный K8s кластер внутри Docker. Это потребляет ресурсы.

Запуск: `minikube start --memory=4096 --cpus=2`

То есть minikube резервирует 4 ГБ RAM. Плюс сами Pod-ы внутри.

| Что работает | RAM |
|---|---|
| minikube (K8s control plane + kubelet) | ~800 МБ |
| backend pods × 3 | ~300 МБ |
| postgres pod | ~100 МБ |
| redis pod | ~50 МБ |
| nginx pod | ~50 МБ |
| **Итого** | **~1.3 ГБ внутри minikube** |

На VPS с 2 ГБ RAM это не запустится нормально. На 4 ГБ — работает.

**На этом уровне Docker Compose уже не нужен** — ты переключился на K8s. Так что нет ситуации где и compose и minikube одновременно.

**1 VPS достаточно при наличии 4 ГБ RAM.**

---

### Уровень 6 — Мониторинг

Самый тяжёлый уровень с точки зрения памяти.

| Сервис | RAM |
|---|---|
| backend × 3 + nginx + postgres + redis | ~380 МБ |
| prometheus | ~200 МБ |
| grafana | ~150 МБ |
| loki | ~150 МБ |
| promtail | ~50 МБ |
| cadvisor | ~100 МБ |
| **Итого** | **~1 ГБ** |

На 2 ГБ VPS с операционной системой (~400 МБ overhead) это на грани. На 4 ГБ — комфортно.

**1 VPS достаточно при наличии 4 ГБ RAM.**

---

### Уровень 7 — Helm

Используется тот же minikube что на уровне 5. Требования идентичны уровню 5.

**1 VPS достаточно при наличии 4 ГБ RAM.**

---

### Уровень 8 — GitOps (ArgoCD)

ArgoCD устанавливается внутрь minikube. Добавляет ~300-500 МБ к потреблению.

**1 VPS достаточно при наличии 4 ГБ RAM.** На 6 ГБ — комфортно.

---

### Уровень 9 — Terraform

В курсе Terraform управляет Docker-контейнерами локально (Docker provider). Это не создаёт облачные серверы — просто управляет контейнерами через Terraform вместо docker-compose.

Если захочешь попрактиковать создание реальной облачной инфраструктуры (EC2, DigitalOcean Droplet, Yandex Cloud) — это потребует аккаунт в облаке и деньги, но не второй VPS — Terraform работает с твоей локальной машины или с основного VPS.

**1 VPS достаточно.**

---

### Уровень 10 — Ansible

Ansible может управлять localhost (`ansible_connection=local`). Для уровней 1-4 это полностью достаточно.

Шаг 9 ("Multi-server") — единственный где нужна вторая машина. Но это опциональный шаг.

**Варианты для Ansible multi-server без второго VPS:**
- Запустить Docker-контейнер как "второй сервер" (`docker run -d --name server2 ubuntu:24.04 /sbin/init`) — Ansible подключится к нему по SSH как к обычному серверу
- Взять временный дешёвый VPS на 1-2 дня специально для этого упражнения

**1 VPS достаточно для 90% уровня.** Для multi-server практики — второй VPS на пару дней или Docker-контейнер.

## Рекомендуемая конфигурация VPS

Нагрузка по курсу не растёт монотонно — она скачкообразная (см. таблицу выше). Поэтому вместо одной цифры "на весь курс" — два профиля по фазам.

### Профиль А — лёгкие уровни (1, 2, 3, 3.5, 4a, 9, 10)
- **1-2 CPU, 2 ГБ RAM, 20 ГБ SSD** — достаточно с запасом
- Уровень 4b (GitLab) сюда не входит — у него отдельные тяжёлые требования, см. предупреждение выше
- Цена: ~$4-6/месяц (Hetzner CX11/CX22, DigitalOcean Basic)

### Профиль Б — тяжёлые уровни (5, 6, 7, 8: Kubernetes control plane, Prometheus + Loki + Grafana)
- **от 4 ГБ RAM**, 2+ CPU, 40 ГБ SSD
- minikube резервирует ~4 ГБ сам по себе (`--memory=4096`), плюс Pod'ы и стек мониторинга поверх — меньше 4 ГБ будет постоянно упираться в OOM
- Цена: ~$12-16/месяц (Hetzner CX32, DigitalOcean 2 vCPU 4 GB)

**Практический вывод:** апгрейд VPS по ходу курса у большинства провайдеров — это пара минут и рестарт, не пересоздание сервера. Начинай с Профиля А, апгрейдь перед Уровнем 5. Если не хочешь думать об апгрейде — бери сразу Профиль Б, он с запасом покрывает обе фазы. Hetzner дешевле DigitalOcean при сравнимом железе.

## Когда реально нужен второй сервер

Три сценария где второй VPS меняет учебный опыт:

### 1. Ты хочешь пройти уровень 4b (GitLab) полностью

Self-hosted GitLab — уникальный опыт: CI/CD под твоим контролем, runner на твоей машине, secrets в твоей базе. Если это важно — отдельный VPS на 4+ ГБ под GitLab (или бери gitlab.com).

### 2. Ты хочешь увидеть реальное горизонтальное масштабирование

На уровне 2 три контейнера на одной машине симулируют scale-out. Ты не увидишь реальной разницы в производительности при добавлении "инстансов" — они делят одно железо.

Если хочешь увидеть настоящий эффект: два VPS, один — балансировщик (Nginx), второй — бэкенд. Добавляешь третий VPS с бэкендом — latency реально падает. Это уже production-ближе.

Но для понимания концепций и конфигурации — один VPS абсолютно достаточен.

### 3. Ansible fleet management (уровень 10)

Управление 5-10 одинаковыми серверами — это то для чего Ansible и создавался. На одной машине ты видишь идемпотентность и механику, но не параллелизм и реальный fleet.

Если хочешь это почувствовать: 2-3 маленьких VPS (можно самых дешёвых, 1 ГБ) на пару дней. Или симулируй через Docker-контейнеры.

## Практические советы по управлению ресурсами

### Следи за памятью
```bash
# Запускай перед стартом нового уровня:
free -h
docker stats --no-stream
```

Если свободной памяти меньше 500 МБ перед запуском уровня — возможно зависание.

### Настрой swap (страховка)
```bash
# Создать 2 ГБ swap (один раз при настройке):
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
```

Swap не заменяет RAM — он медленный. Но он спасает от OOM kill когда чего-то не хватает на секунду.

### Чисти Docker-мусор между уровнями
```bash
# После завершения уровня:
docker compose down -v          # остановить + удалить volumes
docker system prune -f          # удалить неиспользуемые образы и контейнеры
docker volume prune -f          # удалить osirotevshie volumes
```

Образы Docker копятся незаметно. После 5-6 уровней без чистки легко потерять 10-20 ГБ на диске.

### minikube и Docker Compose — не вместе

На уровне 5 ты переходишь в minikube. Docker Compose уровней 1-4 в это время должен быть остановлен. minikube создаёт свой Docker daemon — они не мешают друг другу, но ресурсы делят.

```bash
# Перед стартом уровня 5:
cd ../level-3-caching && docker compose down -v
cd ../level-5-kubernetes
minikube start --memory=4096 --cpus=2
```

## Вывод

**Один VPS хватает на весь курс**, но требования зависят от фазы: 1-2 CPU / 2 ГБ RAM для уровней 1-4 и 9-10 (Профиль А), от 4 ГБ RAM — начиная с Kubernetes (уровень 5) и до конца курса, так как уровни 6-8 работают поверх того же кластера (Профиль Б).

Единственное исключение — уровень 4b (GitLab self-hosted): либо gitlab.com, либо второй маленький VPS.

Всё остальное проходится последовательно: остановил предыдущий уровень, запустил следующий. Ресурсы не накапливаются.
